Budgetting website
The theme that set me on thinking about this pattern.
My idea of application of the classic Hub, Mother Ship or Single Dependency Injection pattern.
The germ of the idea came up when developing the original budgetting application I conceptualized the budgetting site into three main sets of elements
  • Requirements (or needs)
  • Contributions
  • Budget Status
  1. Whenever a requirement was added, the required amount in the budget went up, as also a new update to budget history lists, so we had a dependency injected of the budget status class in the requirements class
  2. Likewise for contributions added, it was DI for budget status.
  3. While with the budget there would potentially be dependency injection from requirements and contributions

Now, if I were to add or change, say, add one more MVC (let's say I want to be able to factor in non cash contributions), or automate something, like the allocation when a new contributor or requirement is added, it could potentially mean having to change multiple files and logic.
(A further example:
I have worked in the manufacturing industry. A product has many many components and other dependencies which keep changing. Imagine trying to maintain this using a network of MVC flows...)
